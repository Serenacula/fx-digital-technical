# Test Suite: Full Upload-to-Chart Pipeline

## What this covers

Cross-cutting validation of the full user journey: from page load through image upload, chart render, and slider interaction. The work-item tests cover each component in isolation; this suite verifies the complete data path — pixel extraction → quantization → chart rendering — end-to-end with the algorithm and processor wired together through the UI. It also covers the pipeline's reset behaviour on re-upload and the error-to-recovery flow.

This suite is not redundant with the individual work-item tests. What it catches that they don't: (1) integration bugs where components are correctly implemented individually but mismatched at the handoff — e.g. the UI passing the wrong `totalPixels` to `sortedColours`; (2) the full quantization chain from raw pixel map through slider-driven reaggregate to rendered chart; (3) state machine interactions that span multiple events (upload → slider → re-upload → error → re-upload).

## Prerequisites

- `0001-astro-scaffold` — project must build and serve
- `0002-colour-algorithm` — algorithm must be implemented and unit-tested
- `0003-image-processor` — processor must be implemented and browser-tested
- `0004-ui-page` — UI must be implemented and integration-tested

## Test scenarios

### Scenario 1: Complete single-image flow with known pixel values

**Setup:** dev server running; `tests/fixtures/2x1-red-blue.png` available (2×1, left: #ff0000, right: #0000ff, totalPixels: 2)
**Steps:**
1. Navigate to the app
2. Upload `2x1-red-blue.png`
3. Wait for results state
4. Read the chart rows from the DOM
5. Move the slider to value 1 (no quantization change expected)
6. Read the chart rows again
**Expect:**
- After upload: 2 rows. Row 1: label `#ff0000`, percentage `50.0%`. Row 2: label `#0000ff`, percentage `50.0%`. Order: either, since counts are equal.
- After slider move to 1: same 2 rows, same percentages (bucketSize 1 → no quantization change)
- No console errors at any step

---

### Scenario 2: Slider quantization merges colours end-to-end

**Setup:** `tests/fixtures/2x1-near-black.png` — 2×1, left: `(10, 0, 0, 255)`, right: `(30, 0, 0, 255)`

Quantization verification (confirmed):
- At bucketSize 10: `quantize(10, 10)` = `Math.round(1.0) * 10` = 10 → key `"10,0,0"` → `#0a0000`; `quantize(30, 10)` = `Math.round(3.0) * 10` = 30 → key `"30,0,0"` → `#1e0000`. Two separate entries.
- At bucketSize 64: `quantize(10, 64)` = `Math.round(0.156) * 64` = `0 * 64` = 0; `quantize(30, 64)` = `Math.round(0.469) * 64` = `0 * 64` = 0. Both merge to key `"0,0,0"` → `#000000`. One entry.

**Steps:**
1. Navigate to the app
2. Upload `2x1-near-black.png`
3. Wait for results (default bucketSize 10)
4. Note `window.__recalcCount` (should be 1)
5. Move slider to value 64; wait 100ms for debounce
6. Read chart rows via DOM

**Expect:**
- After upload at bucketSize 10: 2 rows — `#0a0000` (50.0%) and `#1e0000` (50.0%)
- After slider to 64: 1 row — `#000000`, percentage `100.0%`
- `window.__recalcCount` is 2 (one recalculation for the slider move)
- `window.__createObjectURLCount` is 1 (no image re-read)

### Scenario 4: Error state then successful recovery

**Setup:** dev server running
**Steps:**
1. Navigate to app
2. Upload a GIF file (invalid type)
3. Observe error state
4. Upload `tests/fixtures/1x1-white.png`
5. Observe results state
**Expect:**
- After GIF upload: error message displayed; file input enabled; slider disabled
- After PNG upload: error message cleared; chart shows 1 row with `#ffffff`, `100.0%`; slider enabled
- Image preview shows the white PNG

---

### Scenario 5: All accepted file formats process without error

**Setup:** one valid test image in each of the five accepted formats — `1x1-format.jpg`, `1x1-format.webp`, `1x1-format.avif`, `1x1-format.bmp` (plus PNG already tested). All generated by the fixture script using `sharp`.
**Steps:** for each format fixture:
1. Navigate to app (fresh page load or reload)
2. Upload the fixture
3. Wait for results state
**Expect:** for each upload:
- No ValidationError message in the chart column
- Chart column transitions from "Analysing…" to showing at least one bar row
- `totalPixels` is 1 (confirmed via `window.__lastResult`)
- `map` is non-empty
- File input and slider are re-enabled

_(JPEG exact values are not asserted — lossy. WebP, AVIF, and BMP fixtures are generated lossless and can be asserted exactly — see 0003-image-processor.md Section C.)_

### Scenario 6: Percentage text shows on each bar row

**Setup:** upload `tests/fixtures/2x1-red-blue.png`
**Steps:**
1. Upload image
2. Wait for results
3. Read the text content of each bar row
**Expect:**
- Each row has visible percentage text formatted as `toFixed(1)` — e.g. `"50.0%"`, `"33.3%"`, `"16.7%"`
- The percentage text is accurate: each entry's `count / totalPixels * 100`, formatted to one decimal place
- Percentage text is distinct from the bar element (it's a separate element or text node, not just the bar label)

## Acceptance criteria

All 6 scenarios must pass. Scenario 2's exact quantization math is documented above — the builder should verify the fixture pixel values produce the expected merging behaviour at bucketSize 64 before writing the test assertion. Scenario 5's format tests must pass for all five accepted MIME types.
