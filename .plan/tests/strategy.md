# Test Strategy: Image Dominant Colour Finder

## Philosophy

The system has two distinct halves with very different testing characteristics. The colour algorithm (`src/lib/colour-algorithm.ts`) is pure TypeScript — deterministic, no I/O, no browser dependency — and is the highest-value testing target. Extensive unit testing here gives strong confidence and fast feedback. The image processor and UI are browser-bound, which makes them harder to test in isolation; coverage there is shallower but focussed on the behaviours that would be invisible to algorithm tests: that canvas pixel extraction produces the right frequency map, that UI states transition correctly, and that the full upload-to-chart pipeline works end-to-end.

The critical failure modes are: (1) wrong pixel counts in the frequency map — this would silently produce wrong colours; (2) quantization arithmetic bugs — these would produce wildly wrong output; (3) UI state machine errors — e.g. the chart rendering from a null rawMap, crashing silently. The test plan prioritises catching all three.

## Tooling

- **Unit tests:** Vitest — `environment: 'node'` for `colour-algorithm` (pure functions, no browser dependency); `environment: 'jsdom'` for `image-processor` ValidationError cases (needs the `File` constructor)
- **Browser tests / E2E:** Playwright with Chromium (covers image-processor pixel extraction and all UI flows)
- **Test runner — unit:** `npm run test:unit` (vitest)
- **Test runner — E2E:** `npm run test:e2e` (playwright, with webServer config pointing at `npm run dev`)
- **Test runner — all:** `npm run test` (runs unit first, then E2E)

## Infrastructure

- **Test image generation**: PNG pixel-extraction fixtures are generated once by `scripts/generate-test-fixtures.ts`, which uses `pngjs` (pure-JS, devDependency) to write PNG files from explicit RGBA pixel arrays and verifies each output by decoding it back. Format-compatibility fixtures (JPEG, WebP, AVIF, BMP) are generated by the same script using `sharp` (devDependency). All fixture files are committed to `tests/fixtures/` — the script is not run as part of the test suite, only when fixtures need to be (re-)created. Each fixture's intended pixel values are documented in the script and in the test plan.
- **Dev server**: Playwright's `webServer` config starts `npm run dev` automatically before the E2E suite runs. Tests navigate to `http://localhost:4321/fx-digital-technical/`.
- **No databases, no external services, no environment variables** — everything is client-side.
- **Large file simulation**: Use `new ArrayBuffer(17 * 1024 * 1024)` to simulate an oversized file in validation unit tests without creating a real large file on disk.

## Test hooks (dev mode only)

The following `window.*` properties must be set by the UI `<script>` when `import.meta.env.DEV` is `true`. They must not exist in production builds. The builder defines these in one block in the script.

- **`window.__lastResult: { map: RawMap, totalPixels: number }`** — set immediately after a successful `processImage()` call, before reaggregation. Reflects the raw extracted map. Read by image-processor pixel-extraction tests and results-state assertions.
- **`window.__lastRendered: ColourEntry[]`** — set after each chart re-render, with the final `sortedColours()` output at the current bucketSize. Read by E2E suite scenarios that assert post-quantization output.
- **`window.__recalcCount: number`** — initialised to `0` on page load; incremented each time the render pipeline runs (each time `reaggregate` is called). Read by the debounce test.
- **`window.__delayProcessing: number | undefined`** — if set to a positive number before a file is uploaded, the UI script waits that many milliseconds before calling `processImage`. Makes the transient loading state reliably observable in tests. Default: `undefined`.

## Coverage expectations

- **`colour-algorithm`**: Thorough — every exported function covered, including boundary values and the transparent sentinel at every layer. This is the safest, highest-value testing real estate in the project.
- **`image-processor` (validation)**: Thorough for the ValidationError paths. These are pure conditional checks and easy to verify.
- **`image-processor` (pixel extraction)**: Moderate — several canonical cases using generated images. Not exhaustive over the full RGBA space; the goal is confidence that the extraction loop is wired correctly.
- **`ui`**: Integration-level only. The UI JS is not exported and can't be unit tested in isolation. Playwright covers the full state machine (empty → loading → results → error) and the slider re-render path.
- **`scaffold`**: Light — manual smoke test only. No logic to cover.

## Run order

Unit tests must pass before E2E runs. E2E can be run independently during development by starting `npm run dev` manually, but `npm run test` enforces the order via script chaining.

## CI considerations

CI is out of scope for this project. If added later: unit tests run on every push; E2E runs on PRs targeting main (slower, needs a full build/serve step).
